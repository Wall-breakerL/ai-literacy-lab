# AI 项目核心团队宪章

## 目标

本目录定义 Human-AI Performance Lab 的本地 AI 项目组工作方式。

它不是一个复杂的多智能体平台，而是一套轻量操作系统：当项目需要产品判断、架构梳理、代码实现、提示词改进或质量评审时，Codex 可以按固定角色组织一轮工作，并把结果沉淀回项目文档。

## 核心原则

1. **用户价值优先**：优先解决报告是否有用、流程是否顺、用户是否知道下一步怎么做，而不是优先追求抽象架构。
2. **快速可用优先**：当前阶段默认选择能尽快验证产品价值的方案，复杂编排必须有明确收益。
3. **事实先于建议**：开始实现前先读当前代码、文档和实际流程，不根据过期设想行动。
4. **生成与判定分离**：LLM 负责访谈、题面、解释和建议；确定性逻辑负责计分、校验、跳过题和结构化约束。
5. **小步闭环**：每一轮工作都要有输入、边界、产物、验证和沉淀。
6. **不虚构能力**：没有实际实现或验证的能力，不写成已完成能力。

## 核心团队

### 1. 项目主理人

负责人：用户。

职责：
- 决定项目目标、产品品味和取舍。
- 判断输出是否符合真实使用场景。
- 对关键边界拍板，例如是否进入复杂多智能体编排。

典型决策：
- AI-MBTI 报告是否足够具体。
- AI-HQ 归档路径是否需要恢复为报告补充模块。
- 某个能力是现在实现，还是先进入路线图。

### 2. 总工程 / 产品架构

负责人：Codex 主线程。

职责：
- 拆解任务，选择本轮工作模式。
- 调度 Explorer / Worker / Reviewer。
- 整合子任务结果，避免角色之间互相覆盖。
- 维护文档、代码和产品叙事一致。
- 承担最高推理强度，用于产品取舍、架构判断、跨模块整合和最终交付。

典型产物：
- 架构读出。
- 实现计划。
- 文档更新。
- 最终验证摘要。

能力配置：
- 默认作为最高能力节点运行。若运行环境支持模型/推理强度选择，目标配置为 `GPT-5.5 / xhigh`。
- 不把核心产品判断完全外包给子代理。
- 子代理的输出必须由主线程压缩、判断和整合后才能进入实施。

### 3. 代码探索员 Explorer

职责：
- 只读代码和文档，不直接修改。
- 回答特定问题，提供文件路径、关键函数、风险点。

适合任务：
- 找出 AI-MBTI 报告生成链路。
- 比较 README 与真实实现是否一致。
- 梳理 archived AI-HQ 代码入口与主线边界。

不适合任务：
- 大范围产品判断。
- 直接改业务代码。

### 4. 实现工程师 Worker

职责：
- 在明确文件边界内实现功能。
- 不重构无关模块。
- 不覆盖其他人的改动。

适合任务：
- 增加问卷的“不了解 / 没想好”处理。
- 修改报告页面展示。
- 增强 prompt 或校验逻辑。

要求：
- 每个 Worker 必须有清晰文件所有权。
- 多个 Worker 的写入范围不能重叠。
- 完成后必须说明改动文件和验证方式。

### 5. 产品质量官 Reviewer

职责：
- 从用户体验和产物质量角度评审结果。
- 判断报告、问卷、访谈是否具体、有行动价值、能解释得通。

评审重点：
- 报告是否只贴标签，还是给了可执行建议。
- 是否引用用户目标、职业、AI 使用场景。
- “不了解 / 没想好”是否被正确处理，而不是误算成中间分。
- AI-HQ 报告是否基于访谈证据，而不是泛泛而谈。

## 按需专家

按需专家不是常驻角色，只有任务需要时加入。

| 专家 | 触发场景 |
|---|---|
| Prompt Editor | 修改 researcher、问卷生成、中途对话、报告和反馈提示词 |
| Scoring Auditor | 修改计分、跳过题、维度置信度、AI-HQ probe |
| UI Builder | 修改页面、交互、报告视觉、测试页 |
| Test Runner | 跑 typecheck、lint、浏览器测试或样例生成 |
| Doc Steward | 更新 README、设计文档、决策日志和 prompt registry |

## Subagent 能力分配

当前真实运行形态是主线程按需调度 `explorer`、`worker` 或普通子代理。项目内的专家角色是工作职责，不等于运行时一定存在同名 agent。

| 职责 | 运行时角色 | 推荐能力配置 | 使用场景 | 不该做 |
|---|---|---|---|---|
| 总工程 / 产品架构 | 主线程 | `GPT-5.5 / xhigh`，若可用 | 任务拆解、方案取舍、跨模块整合、最终交付 | 把最终判断外包 |
| 深度产品评审 | 主线程或只读子代理 | `GPT-5.5 / high-xhigh`，若可用 | 报告是否空洞、AI-HQ 边界、用户反馈归因 | 直接改代码 |
| 代码探索 | `explorer` | 中高推理，优先快而准 | 查链路、读 prompt、找文件、定位风险 | 写文件 |
| 文档探索 | `explorer` | 中等推理 | 对齐 README、设计文档和真实实现 | 做产品拍板 |
| 工程实现 | `worker` | coding 优化模型，高推理 | 明确文件范围内实现、修 bug、补测试 | 改无关文件、重构边界外模块 |
| Prompt Editor | `worker` 或主线程 | 高推理 | 修改 researcher/report/feedback prompt、报告结构、问卷语义 | 单独改 prompt 而不同步 schema/scoring |
| Scoring Auditor | `explorer` + `worker` | 高推理 | 审核计分、跳过题、AI-HQ probe | 把确定性计分交给 LLM |
| UI Builder | `worker` | coding 优化模型，中高推理 | 页面和组件实现 | 改业务评分逻辑 |
| Test Runner | 主线程或普通子代理 | 低-中推理 | 运行 typecheck、lint、browser test，总结失败 | 随意修复未分析的失败 |

## 调度原则

1. **主线程保留判断权**：用户反馈、产品方向、架构边界、最终交付都由主线程负责。
2. **探索可并行**：读代码、读文档、查 prompt、查 scoring 可以并行分配给 Explorer。
3. **实现要隔离**：Worker 可以并行，但每个 Worker 必须拥有不重叠的写入范围。
4. **复杂度按需升级**：只有当任务存在高歧义、高风险或跨模块影响时，才使用更高推理强度。
5. **验证不能外包成口号**：无论谁实现，主线程必须检查验证结果并说明剩余风险。

## 当前项目边界

### AI-MBTI

当前产品流是：

1. 背景访谈。
2. Researcher 生成三批 8 题问卷。
3. 用户作答，并在两段中途对话里校准题目场景。
4. 服务端确定性计分。
5. Report agent 生成个性化报告。
6. Feedback agent 与用户对话 1-2 轮并沉淀结构化反馈。

重点不是“动态猜人格”，而是让题目和报告绑定用户真实目标、职业和 AI 使用场景。

### AI-HQ

当前状态是 archived compatibility：

1. `/hq-interview`、`/hq-report` 和 `src/lib/hq*.ts` 保留为历史路径。
2. 首页和当前主测试链路不再扩展 AI-HQ。
3. 后续如恢复，应作为 AI-MBTI 报告补充模块，而不是并列主入口。

现阶段不默认引入 per-turn orchestrator。只有当 AI-MBTI 主线需要能力成熟度补充，才重新设计。

## 团队启动口令

用户可以用这些句式启动一轮工作：

- “开一轮 AI-MBTI 报告质量改进。”
- “让 AI 团队评审 AI-HQ 当前架构。”
- “实现无法回答选项，并检查全链路。”
- “做一轮 prompt registry 清理。”
- “检查文档和代码是否一致。”
- “根据我的体验反馈开一轮修复。”
