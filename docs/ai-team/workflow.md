# AI 项目组运行流程

## 一轮工作的标准结构

每一轮 AI 项目组工作都遵循：

```text
任务输入 -> 边界判断 -> 并行探索 -> 决策草案 -> 实现 -> 验证 -> 沉淀
```

目标是避免“聊了很多但没有产物”，也避免“一上来就乱改代码”。

## 主线程职责

Codex 主线程是总工程 / 产品架构角色，负责最高层判断。

若运行环境支持模型和推理强度配置，主线程目标配置为：

```text
GPT-5.5 / xhigh
```

主线程必须亲自完成：

- 理解用户反馈和真实体验。
- 判断本轮是否需要并行子代理。
- 给每个子代理分配清晰任务边界。
- 整合子代理输出并做产品/架构取舍。
- 决定是否进入工程实施。
- 组织验证并交付最终说明。

主线程不能把这些事情完全交给子代理。

## Subagent 调度矩阵

| 任务类型 | 推荐 subagent | 推荐能力配置 | 是否并行 | 输出 |
|---|---|---|---:|---|
| 读代码链路 | `explorer` | 中高推理，速度优先 | 是 | 文件、函数、事实、风险 |
| 读文档一致性 | `explorer` | 中等推理 | 是 | 过期描述、不一致点 |
| Prompt 诊断 | `explorer` 或主线程 | 高推理 | 是 | prompt 职责漂移、改进点 |
| 计分/校验诊断 | `explorer` | 高推理 | 是 | 确定性逻辑风险 |
| 明确代码实现 | `worker` | coding 优化模型，高推理 | 可并行 | 文件改动、验证说明 |
| UI 修改 | `worker` | coding 优化模型，中高推理 | 可并行 | 页面/组件改动 |
| 产品质量评审 | 主线程或只读子代理 | 高-超高推理 | 通常否 | 是否解决用户体验问题 |
| 测试运行 | 主线程或普通子代理 | 低-中推理 | 可并行 | 命令结果、失败归因 |

并行规则：

- Explorer 可以并行读不同模块。
- Worker 只有在写入文件不重叠时才能并行。
- Reviewer 默认不和实现并行做最终结论，避免基于旧代码评审。
- 主线程必须在实现前后各做一次整合。

## Step 1：任务输入

用户用一句话描述本轮目标。

示例：

- “提升 AI-MBTI 报告的行动价值。”
- “检查 archived AI-HQ 是否还影响当前主线。”
- “把不了解 / 没想好选项做完整。”
- “把 README 和当前实现对齐。”

Codex 主线程需要追问的情况：

- 任务涉及产品方向且存在多个合理路径。
- 改动会影响数据结构、报告格式或用户流程。
- 用户的目标与当前文档或代码事实冲突。

不需要追问的情况：

- 文件边界清晰。
- 用户已明确要实现。
- 可以通过读代码直接确认事实。

## Step 2：边界判断

Codex 主线程先把任务归类。

| 类型 | 是否改代码 | 典型产物 |
|---|---:|---|
| 产品讨论 | 否 | 方案、取舍、下一步 |
| 架构评审 | 通常否 | 风险清单、模块关系、建议 |
| 实现任务 | 是 | 代码、测试、文档 |
| Prompt 改进 | 可能 | prompt、样例、评估标准 |
| 质量评审 | 通常否 | findings、修复建议 |
| 文档沉淀 | 是 | docs / README 更新 |

边界判断必须明确：

- 本轮是否修改文件。
- 可能涉及哪些模块。
- 哪些问题需要用户拍板。
- 哪些验证必须完成。

## Step 3：并行探索

当信息不够时，主线程可以拉起 Explorer 做只读调查。

常见探索分工：

| Explorer | 问题 |
|---|---|
| Flow Explorer | 当前页面/API/状态流怎么走 |
| Prompt Explorer | researcher / report / feedback prompt 在哪里，职责是否漂移 |
| Scoring Explorer | 计分、跳过题、置信度是否一致 |
| Docs Explorer | README 和设计文档是否与代码一致 |

Explorer 输出必须短而具体：

- 相关文件路径。
- 当前事实。
- 风险或不一致点。
- 推荐下一步。

## Step 4：决策草案

主线程把探索结果压缩成可决策内容。

格式：

```text
当前事实：
- ...

主要问题：
- ...

推荐方案：
- ...

需要用户拍板：
- ...
```

如果用户已经授权实现，且方案边界清晰，主线程可以直接进入实现。

## Step 5：实现

实现可以由主线程完成，也可以由 Worker 完成。

Worker 使用规则：

- 每个 Worker 只负责一块明确文件范围。
- 多个 Worker 不能写同一文件。
- Worker 不做无关重构。
- Worker 不能回滚未归属自己的改动。

推荐写入边界示例：

| 任务 | 文件边界 |
|---|---|
| AI-MBTI prompt 改进 | `src/lib/researcher.ts`, `src/lib/reportAgent.ts`, `src/lib/feedbackAgent.ts` |
| 问卷兜底与校验 | `src/lib/fallbackQuestionnaire.ts`, `src/lib/questionnaireValidation.ts` |
| 报告计分 | `src/lib/reportScoring.ts` |
| archived AI-HQ 评分 | `src/lib/hqScoring.ts`, `src/lib/hqAgents.ts` |
| 报告页 UI | `src/app/report/page.tsx`, `src/components/*` |
| 文档更新 | `README.md`, `docs/**` |

## Step 6：验证

默认验证清单：

- `npm run typecheck`
- `npm run lint`
- `npm run test:browser`，当页面或核心逻辑测试相关时运行

产品质量验证：

- AI-MBTI 报告是否引用用户目标或最近使用场景。
- 报告是否包含整体建议、行动建议和可直接使用的 prompt 模板。
- “不了解 / 没想好”是否跳过计分。
- 低有效题数是否降低置信表达，而不是强行下结论。
- archived AI-HQ 是否仍被清楚标记为非主线，不误导成当前能力。

如果验证不能运行，必须说明原因和剩余风险。

## Step 7：沉淀

每一轮重要工作至少更新一个沉淀位置。

| 内容 | 文件 |
|---|---|
| 产品或架构决策 | `docs/ai-team/decision-log.md` |
| 报告/访谈质量标准 | `docs/ai-team/evaluation-rubric.md` |
| Prompt 变更 | `docs/ai-team/prompt-registry.md` |
| 流程或角色变化 | `docs/ai-team/workflow.md` |
| 面向用户的产品说明 | `README.md` |

沉淀内容必须区分：

- 已实现。
- 已决定但未实现。
- 待讨论。
- 已废弃。

## 推荐轮次模板

### 产品评审轮

```text
目标：评审一个产品方向或功能价值。
产物：问题清单、推荐方案、需要拍板的点。
默认不改代码。
```

### 实现轮

```text
目标：完成一个明确功能或修复。
产物：代码改动、验证结果、必要文档更新。
默认改代码。
```

### 报告质量轮

```text
目标：提升 AI-MBTI 报告可用性；AI-HQ 仅在归档兼容问题出现时处理。
产物：样例报告评审、prompt 或 scoring 改动、质量标准更新。
```

### 文档对齐轮

```text
目标：让 README、设计文档和真实实现一致。
产物：文档 diff、过期描述删除、当前流程说明。
```

### 反馈驱动交付轮

```text
目标：根据用户真实体验反馈，完成研究、方案、实施、自检和交付。
产物：问题归因、改进方案、代码/文档改动、验证结果、下一轮体验点。
默认先诊断，再实现。
```
