# AI 项目组质量评估标准

## 使用方式

每次完成产品、prompt、报告、UI 或实现改动后，用本文件做快速验收。

评分不需要复杂量化。每项标记为：

- `Pass`：符合要求。
- `Weak`：可用但质量不足。
- `Fail`：会误导用户或影响核心体验。
- `N/A`：本轮不相关。

## AI-MBTI 访谈质量

| 项目 | Pass 标准 |
|---|---|
| 背景收集 | 收集职业/身份、AI 工具或最近使用场景 |
| 目标上下文 | 尽量获得当前目标；缺失时能用 recentUse 降级 |
| 访谈边界 | 背景阶段不直接追问维度或诱导类型 |
| 自然程度 | 语气像轻量访谈，不像审讯或表格填写 |
| 转问卷时机 | 约 2 轮后进入问卷，避免拖长 |

## AI-MBTI 问卷质量

| 项目 | Pass 标准 |
|---|---|
| 数量 | Phase 6 主链路固定 24 题，两部分 8+16 |
| 覆盖 | Relation / Workflow / Epistemic / RepairScope 第一部分各 2 题、第二部分各 4 题，全流程各 6 题 |
| 场景绑定 | 题干绑定用户职业、AI 工具、最近任务或目标 |
| 批次结构 | `hybrid_batch1` 为 4 道习惯题 + 4 道场景题；`hybrid_batch2` 为 8 道习惯题 + 8 道场景题 |
| 正反向题 | 第一部分每维 2 正向 + 0 反向；第二部分每维 2 正向 + 2 反向；两部分合计每维 4 正向 + 2 反向 |
| 跳过选项 | “不了解 / 没想好”独立存在，不进入分数 |
| 可回答性 | 题目不要求用户拥有不存在的经验 |
| 中途校准 | 第一部分后一轮 mid-dialogue 能把用户修正写入 `scenarioGuidance` / `refinedTargetContext` |

## AI-MBTI 报告质量

| 项目 | Pass 标准 |
|---|---|
| 总结 | 不是空泛性格标签，能概括用户 AI 协作方式 |
| 证据 | 至少引用问卷回答或访谈背景中的具体信息 |
| 维度解释 | 解释每个分数为什么合理，而不是复述定义 |
| 目标相关 | 建议能回到用户目标、职业或近期任务 |
| 行动价值 | 给出 2-3 条下一步行动建议 |
| Prompt 模板 | 提供 1-2 个可直接使用的 prompt |
| 置信表达 | 有效答案少时标记为初步观察 |
| 语气 | 像专业测评报告，不像营销文案 |

## AI-HQ 访谈质量（Archived）

| 项目 | Pass 标准 |
|---|---|
| 顺序 | 固定 5 段：Route、Frame 1、Frame 2、Workflow、Repair |
| 聚焦 | 每段只测对应能力，不发散成普通聊天 |
| 证据 | 保留用户原始回答供报告阶段判断 |
| 时长 | 总体保持 5 分钟左右的轻量体验 |

## AI-HQ 报告质量（Archived）

| 项目 | Pass 标准 |
|---|---|
| 分数 | Route / Frame / Workflow / Repair 与探针一致 |
| 等级 | L1/L2/L3 同时满足总分和维度门槛 |
| 证据 | 每个低分或高分判断能回到用户回答 |
| 建议 | 建议对应具体能力短板，不只是泛泛鼓励 |
| 边界 | 不把 AI-HQ 写成通用人格测评 |

## 工程质量

| 项目 | Pass 标准 |
|---|---|
| TypeScript | `npm run typecheck` 通过 |
| Lint | `npm run lint` 通过 |
| 测试 | 相关时 `npm run test:browser` 通过 |
| 数据结构 | prompt、schema、UI、报告使用同一套字段语义 |
| 回退路径 | LLM 输出不合格时有本地 fallback 或校验 |
| 文档同步 | 用户可见流程变化同步到 README 或 docs |

## 红线

以下情况不能标记为完成：

- 报告只输出类型名称，没有行动建议。
- 把“不了解 / 没想好”当作中间分计入。
- 文档声称某能力已实现，但代码没有对应实现。
- 评分完全交给 LLM，缺少确定性校验。
- 把 AI-HQ archived 路径写成当前主线能力，或在没有明确问题证据时重新引入复杂 per-turn orchestration。
