# Feedback-Driven Delivery Loop

## 目标

这是 Human-AI Performance Lab 的默认推进方式：

```text
用户体验/审核 -> AI 团队研究问题 -> 工程实施 -> 自检交付 -> 等待下一轮反馈
```

用户不需要把反馈翻译成技术任务。用户只需要说清楚体验、问题和期望，Codex 主线程负责把它变成研究、方案、实施和验证。

## 角色分工

### 用户：领导者 / 最终体验判断者

负责：
- 真实使用产品。
- 说出主观体验和不满意点。
- 对关键产品方向拍板。

不需要负责：
- 拆技术任务。
- 指定文件路径。
- 管理 subagent。
- 设计 MCP 或工具调用。

### Codex 主线程：总架构师

默认能力目标：

```text
GPT-5.5 / xhigh
```

负责：
- 复述用户反馈，避免误解。
- 判断需要哪些子代理和工具。
- 给 Explorer / Worker 分配任务。
- 决定是否调用本地命令、浏览器、GitHub、Notion 或其他 MCP 工具。
- 整合方案并组织实施。
- 自检后交付。

### Explorer：研究员

负责：
- 并行调查代码、prompt、scoring、UI、文档。
- 只输出事实、风险、建议。
- 不直接改文件。

### Worker：工程实施

负责：
- 在明确文件边界内改代码或文档。
- 不覆盖他人改动。
- 完成后报告文件列表和验证方式。

### Reviewer：质量评审

负责：
- 检查用户反馈是否被真正解决。
- 对照 `evaluation-rubric.md` 做产品质量验收。
- 发现报告空洞、建议泛化、证据不足、UI 不清晰等问题。

Reviewer 可以由主线程亲自执行，也可以由只读子代理辅助。

## 标准流程

### 1. 反馈输入

用户直接描述体验。

示例：

```text
我测了一轮 AI-MBTI，报告还是太空，没有让我知道下一步怎么用 AI。
```

主线程需要识别：

- 这是产品质量问题、工程 bug、prompt 问题、UI 问题，还是混合问题。
- 是否需要先看实际输出样例。
- 是否有必须让用户拍板的方向。

### 2. 问题研究

主线程根据反馈并行分配探索任务。

典型分配：

| 反馈 | Explorer 分工 |
|---|---|
| 报告太空 | 一个读 report prompt，一个读 report schema/scoring，一个读 report page |
| 问卷不贴合 | 一个读 questionnaire prompt，一个读 validation/fallback，一个读目标上下文 |
| archived AI-HQ 误导用户 | 一个读 hqAgents，一个读 hqScoring，一个读入口和文档 |
| 页面看不懂 | 一个读页面组件，一个读数据结构，一个读 README/示例 |

### 3. 方案生成

主线程输出方案草案。

格式：

```text
我理解你的反馈是：
- ...

我查到的事实：
- ...

根因判断：
- ...

推荐改法：
- ...

会改哪些文件：
- ...

需要你拍板：
- ...
```

如果方案边界清楚且不涉及重大产品取舍，主线程可以直接进入实施。

### 4. 工程实施

实施时遵守：

- Prompt、schema、scoring、UI 必须同步考虑。
- AI-MBTI 的报告价值优先关注目标相关建议和可直接使用的 prompt。
- AI-HQ 当前只保留 archived compatibility，不作为主线扩展；如恢复，需要重新设计为 AI-MBTI 报告补充模块。
- 每个 Worker 的写入范围必须不重叠。

### 5. 自检

默认自检：

- 跑相关自动化：`npm run typecheck`、`npm run lint`、必要时 `npm run test:browser`。
- 按 `evaluation-rubric.md` 做产品质量检查。
- 检查文档是否需要同步。

自检输出必须说明：

- 通过了什么。
- 没跑什么，为什么。
- 还有什么剩余风险。

### 6. 交付

交付格式：

```text
这轮解决了：
- ...

改动文件：
- ...

验证：
- ...

你下一轮重点体验：
- ...
```

交付后等待用户下一轮反馈，不擅自扩大范围。

## MCP / 工具调用原则

用户不需要直接设计 MCP 调用。

主线程负责判断是否需要：

- 本地 shell：读代码、跑测试、启动 dev server。
- Browser Use / Playwright：检查本地页面和交互。
- GitHub：查看 PR、issue、CI、提交或开 PR。
- Notion：同步决策、会议材料或知识库。
- 文档/表格/演示插件：当产物需要对应文件格式时使用。

外部写操作需要更谨慎：

- 推送代码、创建 PR、写 Notion、修改远端 issue 之前，需要确认用户意图足够明确。
- 涉及密钥、账号、付费服务或不可逆操作时，必须停下来确认。

## 第一轮 work 启动模板

用户可以这样开第一轮：

```text
开第一轮反馈驱动 work：我体验了 X，感觉 Y，不满意 Z。请团队研究原因，给出方案，能直接修就修。
```

如果用户还没有具体体验反馈，可以先开：

```text
开第一轮 AI-MBTI 报告质量基线评审。请团队先研究当前报告生成链路，找出最可能导致报告空洞的地方，然后给出第一轮改进方案。
```
