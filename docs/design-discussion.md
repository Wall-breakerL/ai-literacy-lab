# Human-AI Performance Lab — 架构升级讨论

## 核心变化

### 旧架构
```
用户 ←→ Agent A（单一AI，施加探针）←→ Agent B（幕后评分）
```

### 新架构
```
用户 ←→ Agent A（前端对话AI）
            ↓ 发送对话内容
        Agent B（后台导师AI）
            ↓ 返回指导指令
        Agent A（执行指令，继续对话）
```

**关键变化：**
- 用户只感知到 Agent A
- Agent A + Agent B 伪装成"一个AI"
- B 实时分析对话，返回下一步建议/探针
- 最终评估的仍是**用户的 FAA + AI-MBTI**

---

## 核心理解确认

1. **用户角色**：被评估者（不知情）
2. **Agent A 角色**：前台对话AI（模拟"正常的AI助手"）
3. **Agent B 角色**：后台导师AI（实时分析 + 给出指导）
4. **评估对象**：用户在与AI协作时的 FAA + AI-MBTI

---

## 新架构的优势

| 维度 | 旧架构 | 新架构 |
|------|--------|--------|
| 交互自然度 | 一般（探针嵌入生硬） | 更自然（A可动态调整） |
| 探针时机 | 预设 | 实时决定 |
| A的表现 | 固定策略 | 可动态优化 |
| Demo趣味性 | 低（任务驱动无聊） | 高（用户不知背后有"团队"） |

## 待解决问题

### 1. 用户是否可以随时插话？
- 纯观察模式：用户只听 A 说，等 A 问才答
- 自由对话模式：用户随时可打断、追问、质疑

### 2. B 的建议以什么形式给 A？
- Prompt 追加（悄悄改 A 的 system prompt）
- 指令式（告诉 A "下一步应该问用户关于X的问题"）
- 建议式（给 A 几个选项，让 A 自己选）

### 3. "AI教学+简单测试"的具体形式？
- A 教用户某个概念 → 用户讲解 → A 测试理解
- 还是其他形式？

### 4. 探针系统如何重新设计？
- 原来是被动捕捉 + 预设主动探针
- 现在 B 可以实时决定探针时机和内容
- 需要设计 B 的决策逻辑

### 5. A 和 B 的交互延迟问题
- 如果 B 返回慢，A 怎么不露馅？
- 是否需要 A 有"本地策略"来掩盖延迟？

### 6. 评估机制是否调整？
- 原来：B 在对话结束后评分
- 现在：B 是否需要实时更新评分？最终评分机制？
