# V5 核心设计：AI-MBTI 面试评估系统

## 1. 核心定位
纯粹的“AI使用风格/性格（AI-MBTI）”分析工具。
**移除**了强调实操能力的 FAA 维度，完全聚焦于用户的认知、习惯、偏好和态度。

## 2. 交互场景：AI 面试
- **形式**：用户与 Agent A 进行一对一的对话面试。
- **透明度**：用户知情（明确知道这是在评估自己的 AI 使用风格）。
- **角色设定**：
  - **Agent A（面试官/引导者）**：负责提出情景问题、追问细节、挖掘用户真实习惯。
  - **Agent B（分析团队/观察者）**：在后台实时分析用户的回答，提取 MBTI 信号，并在面试结束后生成分析报告。

## 3. 评估模型 (AI-MBTI)

## 4. 面试流程设计：动态访谈 (Dynamic Interview)

### 核心机制
- **访谈为主，场景为辅**：单刀直入探讨 AI 使用习惯，不构建庞大的虚拟任务。只在需要时（如测试容错态度）引入轻量级的职业相关假设。
- **动态适配**：第一个问题由用户自由输入身份/职业，后续提问完全基于该身份动态生成。
- **目标驱动**：Agent A 内部维护一个“题库/考察点清单”，必须在 3-6 分钟的对话中，覆盖 AI-MBTI 的 4 个核心维度。

### Agent A 与 B 的协作难题与解法
**难点**：如何保证自然聊天的同时，不遗漏任何一个维度的考察（覆盖度），并且 Agent B 能准确抓取信号？

## 5. 探针与覆盖策略 (Agent B 驱动逻辑)

### B 的核心职责（主控大脑）
1. **监听与分析**：分析用户最新的回答，提取信号，并判断信号强度（强/弱）。
2. **状态维护**：记录 4 个维度的覆盖状态（未覆盖 / 弱覆盖 / 已覆盖）。
3. **指令下发**：根据状态，向 A 下发下一轮提问指令（如：追问细节、切换新维度）。

### 覆盖判定原则（动态判断）
- **强信号**：用户明确表达了偏好（如“我坚决不信AI说的话”），该维度标记为**已覆盖**，停止该维度追问。
- **弱信号/模糊信号**：用户回答模棱两可（如“有时候信有时候不信”），该维度标记为**弱覆盖**，B 会要求 A 进行**追问**或给出极端的**二选一假设**。
- **未覆盖**：尚未触及的维度，B 会要求 A 引入新话题。

### 信号-维度映射示例
| 信号内容示例 | 对应维度 | 倾向 |
|---|---|---|
| “我直接要代码，不管原理” | Relation | Instrumental (工具型) |
| “我会和它探讨架构设计的合理性” | Relation | Collaborative (伙伴型) |
| “我先想好提纲，让它按点写” | Workflow | Framed (框架型) |
| “我随便扔个需求，看它写什么再改” | Workflow | Exploratory (探索型) |
| “它写的正则我必跑测试” | Epistemic | Auditing (审计型) |
| “一般直接复制就用了” | Epistemic | Trusting (信任型) |
| “写错一点我就重开对话” | Repair Scope | Global (全局型) |
| “哪里错了我就指出来让它改哪里” | Repair Scope | Local (局部型) |


## 6. Agent A 与 B 的通信协议

### Agent B 输出格式 (JSON)
Agent B 每轮评估后输出下一轮的指导方针，**不生成具体话术**，由 A 负责自然语言表达。
```json
{
  "analysis": {
    "signals_detected": [{"dimension": "Relation", "strength": "strong", "tendency": "Instrumental"}],
    "current_status": "Relation 已覆盖，Workflow 弱覆盖"
  },
  "directive": {
    "action": "probe_new" | "follow_up" | "conclude",
    "target_dimension": "Workflow",
    "hint": "用户刚才提到写代码，问他如果是写大项目，是先让AI给框架还是自己写框架让AI填空。"
  }
}
```

### Agent A 输入格式
Agent A 接收用户的回复，以及 Agent B 的 `directive`，结合当前对话上下文，生成一句自然、符合面试官口吻的回复。

## 7. 最终分析报告 (Profile)

面试结束后，Agent B 汇总所有信号，生成两层结构的报告。

### 7.1 概览视图 (Overview)
- **四维雷达图/百分比**：直观展示 Relation, Workflow, Epistemic, Repair Scope 的倾向值。
- **总体评语 (Summary)**：一两句简练的总结（如：“你是一位极具探索精神的 AI 审计官”）。
- **特征标签 (Tags)**：提炼 3 个核心词。

### 7.2 维度详情页 (Detailed Analysis & Advice)
针对每个维度的深度解析，需包含：
1. **倾向判定**：该维度所处的位置。
2. **对话回溯 (Evidence)**：直接引用面试中用户的**原话**，作为判断依据。
3. **专家分析**：解释这句话背后反映的认知习惯。
4. **进阶建议 (Advice)**：
   - **Tone (基调)**：专家感 + 极强亲和力（如：“我们注意到你习惯于... 这个习惯非常高效！不过在面临 XX 情况时，或许可以尝试...”）。
   - **避免**：说教感、居高临下的“你应该如何做”。
   - **强调**：作为 AI 助手，以客观、赞赏且具建设性的口吻给出建议。

## 8. 对话生命周期与结束条件

### 结束条件判定
**对话何时结束完全由 Agent B 决定。**

Agent B 在每轮分析后，检查以下两个条件：
1. **理想结束**：4 个维度（Relation, Workflow, Epistemic, Repair Scope）都已达到**强覆盖**状态。
2. **强制结束 (兜底)**：对话轮数达到上限（暂定 8 轮 / 用户回复次数），防止面试无限拖延。

### 结束流程
1. 当满足任一结束条件时，Agent B 的 `directive` 输出 `{"action": "conclude"}`。
2. Agent A 接收到 `conclude` 指令，生成一段自然友好的结束语（例如：“感谢你的分享，我对你如何使用 AI 已经有了很清晰的了解！稍后会为你生成一份专属分析报告...”）。
3. 前端检测到对话结束状态，触发“生成报告”动画，随后展示最终的 Profile 界面。

## 9. 页面结构与 UI/UX 规划

系统分为三个核心页面，单向流转：

### Page 1: 标题页 (Landing Page)
- **核心目的**：介绍测试目的，获取用户前置身份。
- **元素**：
  - 测试标题与简介（偏向轻松有趣的 AI 性格测试基调）。
  - 一个简单的身份输入框（如：“在开始前，请用一句话描述你的职业或身份：___”）。
  - “开始测试”按钮。

### Page 2: 测试页 (Interview Page)
- **核心目的**：沉浸式对话体验。
- **元素**：
  - **核心区域**：类似微信/ChatGPT 的对话气泡界面。
  - **辅助元素 (轻量级)**：
    - **面试官状态感知**：例如 Agent A 的“正在输入中...”状态。
    - **非侵入式进度反馈**：由于是动态结束，无需显示精确的“第几题/共几题”，但可以设计一个模糊的进度指示器（如：通过颜色变化、或者 Agent B 后台分析的状态灯闪烁，暗示系统正在“懂你更多”）。

### Page 3: 报告页 (Report Page)
- **核心目的**：展示深度分析结果。
- **触发机制**：测试页 Agent A 说出结束语后，界面自动过渡（或提供按钮跳转）至本页。
- **结构**：
  - **概览区**：MBTI 百分比雷达图 / 总体评语 / 特征 Tags。
  - **详情区**：四个维度的深度展开，包含：维度定义、倾向判定、原话引用证据、亲和力的专家建议。
