你现在不是来附和，而是来当“测评方法顾问 + 产品化框架设计师”。

请基于下面背景，重构 AI-HQ（Harness Quotient）Phase 2，使其从“系统架构映射”转成“普通用户在 human-AI task loop 中可观察的控制动作”。

先遵守这些硬要求：

1. 不要把 Harness Engineering 六层直接当成用户能力维度。
   你可以保留原有六维的诊断价值，但必须把它们重写成“用户行为层面可观察的控制动作”。

2. 采用“4个宏域 + 6个诊断子维度”的方案：
   - 委托设计：Frame（任务契约，原 Scope）、Route（执行配置，原 Equip）
   - 过程控制：Plan（拆编，原 Sequence）、State（控境，原 Context）
   - 质量判断：Check（核验，原 Verify）
   - 异常恢复：Repair（复原，原 Recover）

3. 明确解释：
   - 为什么 Scope 必须改写成 AI-specific 的“任务契约/委托设计”，而不是一般表达能力
   - 为什么 Equip 不能只理解为“会不会很多工具”，而要包含“何时不用 AI / 何时加文档、Web、代码、人工复核”
   - 为什么 Route / State / Repair 不能因为缺少暴露机会就直接判低分
   - 为什么 Verify / Check 应该成为高阶等级的重要 gate 维度

4. 评估方式不能是纯自述问卷。
   必须设计为“结构化行为访谈 + 标准化情境题 + 即时诊断题”的混合方案。
   仍然允许是对话式，但不能只是问‘你平时怎么用 AI’。

5. 评分模型从“总分 + 最低子维度”改成“总分 + 最低宏域 + 证据充足度”。
   你必须设计 N/A / insufficient evidence 规则，避免把“没机会展示”误判成“能力低”。

6. 输出必须是可落地的产品设计稿，不要只讲原则。

请按下面结构输出：

A. 先批判现有模型
- 哪些地方方向是对的
- 哪些地方抽象层级错了

- 为什么这样改更 user-centered

C. 对 6 个子维度分别展开
每个维度都要给出：
- 1句定义
- inclusion boundary（算什么）
- exclusion boundary（不算什么）
- 3个最有区分度的高分行为信号
- 3个常见伪信号/误判风险
- 2个 past-behavior interview questions
- 1个 situational vignette
- 1个即时诊断题（例如：给一个缺陷 prompt / 有问题的 AI 输出 / 一个长任务场景）
- 0~3分行为锚定评分标准（BARS）

D. 给出四级成熟度画像
- L1 新手
- L2 进阶
- L3 精通
- L4 专家

要求：
- 写成“实际行为画像”
- 不要写成抽象人格标签
- 要体现四级在委托、控制、判断、恢复上的差异

E. 设计完整访谈协议
至少包含：

F. 设计评分系统
至少包含：

- 宏域怎么 gate
- insufficient evidence 怎么处理

- 哪些维度更适合 soft signal
- 如何避免“会说的人高分、会做的人低分”

G. 给出两个版本
- MVP 版：最小可用、适合快速上线
- Full 版：完整版、适合后续验证与迭代

H. 最后输出“最需要先验证的 5 个风险点”
每个风险点都要给：
- 为什么它会威胁信效度

风格要求：

- 要像一个真正要进入产品设计和测评实施的方案
- 要敢于删改原方案，而不是尽量保留原词
- 中文输出，表格尽量清晰
