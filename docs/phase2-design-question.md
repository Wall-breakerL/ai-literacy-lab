# AI-HQ Phase 2 设计问题

## 背景

我们正在设计一个 AI 能力评估系统，分两个阶段：

**Phase 1 — AI-MBTI**：测用户与 AI 协作的风格偏好，四个维度（Relation / Workflow / Epistemic / RepairScope），通过对话访谈 + 问卷打分，输出风格画像。已完成。

**Phase 2 — AI-HQ（Harness Quotient）**：测用户驾驭 AI 的能力成熟度，对应 Harness Engineering 六层架构，输出 L1新手 / L2进阶 / L3精通 / L4专家 四个等级。等级判定采用总分 + 最低维度双门槛机制。

> **与本文档的关系（工程对齐）**：下文中的六维度、L1–L4 属于设计讨论与目标形态。当前仓库内的 **AI-HQ v0** 为简化实现：**Route / Frame / Workflow / Repair** 四个维度、等级 **L1–L3**（5 轮访谈 + `/api/hq-report`），详见 `src/lib/hqAgents.ts` 与 `src/lib/types.ts`。后续迭代可收紧文档叙事，或扩展实现以向六维/L4 靠拢。

---

## Harness Engineering 背景

Harness Engineering 的核心公式：Agent = Model + Harness。模型是 CPU，Harness 是操作系统。六层架构：

- L1 信息边界层：Agent 该知道什么、不该知道什么
- L2 工具系统层：Agent 怎么跟外部世界交互
- L3 执行编排层：多步骤任务怎么串起来
- L4 记忆与状态层：长任务中间结果怎么管
- L5 评估与观测层：Agent 怎么知道自己做对了没有
- L6 约束、校验与恢复层：出错了怎么办

---

## AI-HQ 六维度设计（当前版本）

| 维度 | 对应 Harness 层 | 核心问题 |
|-----|--------------|---------|
| Scope（定界） | L1 | 能否清晰描述任务结构，明确 AI 的角色与输入边界 |
| Equip（配置） | L2 | 能否选择合适的工具/接口/执行方式 |
| Sequence（编排） | L3 | 能否把长任务拆解为合理的步骤序列 |
| Context（控境） | L4 | 能否管理信息投放节奏、控制上下文利用率 |
| Verify（核验） | L5 | 能否建立独立验证机制、判断完成质量 |
| Recover（恢复） | L6 | 出问题时能否有效纠正、重组或重启 |

---

## 当前遇到的核心问题

**问题一：维度定义的合理性**

六个维度直接从 Harness 六层架构映射而来，但 Harness 六层是系统设计者视角（如何构建 AI 系统），而我们测的是普通用户视角（如何使用 AI）。这两个视角之间的映射是否合理？每个维度在用户行为层面是否真实可观测？

**问题二：Scope 维度的定义争议**

原始定义是"能否清晰定义任务边界、排除无关信息"。讨论后重新定义为"能否清晰描述任务结构，明确 AI 的角色与输入边界"——本质是任务描述能力，背后是对任务结构的认知清晰度。但这个能力是否足够 AI-specific？还是更接近通用的语言/认知能力？

**问题三：维度之间的边界模糊**

- Scope（任务开始前的结构化）和 Context（任务进行中的信息管理）有重叠
- Equip（工具选择）在很多用户场景下可能根本不涉及（只用一个 ChatGPT），这个维度是否对所有用户都有区分度？
- Recover（错误恢复）依赖用户有过失败经历，没有失败经历的用户无法产生有效信号

**问题四：评估方式的信效度**

这六个维度测的是"实际操作中的行为模式"，但访谈只能拿到"用户对自己行为的描述"。自述和实际行为之间存在差距。纯对话访谈能否有效区分真实能力水平？

---

## 我们的问题

1. 这六个维度的设计是否合理？有没有需要合并、拆分或替换的维度？
2. 每个维度在用户行为层面，最有区分度的信号是什么？
3. 通过对话访谈（而非实际操作观测）评估这些能力，有哪些设计原则可以提高信效度？
4. 是否有更好的框架来描述"普通用户驾驭 AI 的能力成熟度"？
