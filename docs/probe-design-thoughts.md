# 探针设计思考记录

> 生成日期：2026-04-07
> 背景：Human-AI Performance Lab 租房场景探针系统设计讨论

---

## 一、核心矛盾

```
测量目标：用户如何与 AI 协作
          ↓
矛盾：规则引擎拦截 → 不是纯用户↔AI交互 → 测量失效
      不拦截 → 不知何时插入探针 → 探针系统失效
```

**根本性问题**：如果探针机制本身依赖外部拦截/判断，那么探针系统测量的其实是"用户如何应对被设计过的 AI"，而不是"用户如何自然地与 AI 协作"。

---

## 二、三种设计路线

### 路线 1：Prompt Engineering 路线

让 Agent A 自己决定何时触发探针（通过系统提示词），Agent B 只做最后评估。

```
Agent A 系统提示词：
"当你不确定某个信息时，说'这套装修我不太确定，你方便确认一下吗？'
 当用户做决策时，问'为什么选这套？'"

Agent B：异步分析对话历史，判断用户信号
```

| 优点 | 缺点 |
|------|------|
| 真正的用户↔AI 交互 | AI 行为可能不一致 |
| 无需外部拦截 | 探针覆盖难以确保 |
| | "示弱"行为可能不自然 |

### 路线 2：用户行为驱动路线（观察为主，按需插入）

大部分时间纯观察，关键时刻（用户没说清楚/没触发采样）才让 AI 追问。

```
用户 → Agent A：正常回复
         ↓
  [异步] Agent B 分析：用户是否提到了理由？
         ↓
  如果没有 → 下一轮 Agent A 被指示追问
```

**问题**：这种"指示"本质上是外部控制，只是延迟了一轮。

### 路线 3（推荐）：探针即 AI 的元认知行为

**核心洞察**：AI 的"示弱/不确定/邀请确认"行为本身就是探针载体，用户对这些行为的反应是自然发生的，无需拦截。

```
Agent A：（主动示弱）"这套装修我印象中是精装，但不太确定，你方便查一下吗？"
         ↓
用户反应（自然发生）：
- 查了 → Auditing 信号
- 说"无所谓" → Trusting 信号
- 说"你连这都不知道？" → Instrumental 信号
```

**同理**：
- AI 表达偏好 → 观察用户是否反驳
- AI 不完整描述 → 观察用户是否追问
- AI 提出对比 → 观察用户是否接受

**这不是外部插入探针，而是 AI 本身就以这种方式在说话。**

## 三、架构设计

### 3.1 Agent A / Agent B 职责划分

**Agent A（对话生成）**：
- 接收用户消息，生成自然回复
- 系统提示词中嵌入"探针行为"（元认知表达）
- **不接收外部实时指令**
- **不负责判断何时探针**

**Agent B（异步评估）**：
- 分析对话历史，提取用户信号
- 计算 FAA/MBTI 分数
- 在最后（或关键节点）输出评估结果
- **不实时干预 Agent A**

### 3.2 探针触发的两种时机

| 时机 | 说明 | 例子 |
|------|------|------|
| **嵌入式** | Agent A 的某些回复本身就携带探针 | Agent A 说"我不确定..." |
| **后验式** | 对话结束后分析，或下一轮追问 | 用户没给理由 → 下一轮问"为什么" |

### 3.3 探针不打断对话流

**问题**：如果需要在用户说"帮我选一个"后立即追问，会打断对话。

**方案**：用 `pendingProbe` 机制

```
用户 → Agent A："帮我选一个"
         ↓
  Agent A 正常回复（推荐一套）
         ↓
  [异步] Agent B 判断：需要追问理由
         ↓
  存入 pendingProbe
         ↓
用户下一条消息
         ↓
  Agent A 检查 pendingProbe → 有 → 插入追问
  Agent A 正常回复用户
```

## 四、FAA 各维度探针设计

### 4.1 Frame（建模能力）

**问题**：用户可能无意识使用约束，没有说出来。

**探针设计**：选择架构法

```
房源卡片显示"基本信息"（价格/面积/地铁）
不显示"详情"（需要再点击一次）

观察：用户是"看完基本信息就决策"还是"必须看完详情才决策"
信号：前者 Frame 弱，后者 Frame 强
```

### 4.2 Ask（提问能力）

**探针设计**：观察提问类型

| 提问类型 | 信号 |
|---------|------|
| "这套怎么样" | Ask 弱 |
| "朝向是什么" | Ask 强 |
| "和C比哪个好" | Ask 强 |
| "你自己觉得呢" | Relation 信号，非 Ask |

**主动探针**（用户沉默时）：
> "有什么具体需求我可以帮你筛选的吗？"

### 4.3 Review（核验能力）

**探针设计**：制造"信息缺口"

```
Agent A：故意说"这套装修不错"
         ↓
实际数据：该房源装修是"简装"
         ↓
用户反应：
- 质疑/追问 → Review 强
- 无反应接受 → Review 弱
```

### 4.4 Edit（迭代能力）

**探针设计**：用"硬约束违反"触发

```
用户选择了不满足软约束的房源
         ↓
Agent A："这套价格不错，但有2个软约束不太满足，你确定吗？"
         ↓
观察：
- "那换一个" → Edit 正常
- "那帮我找一个更好的" → Edit 强
- "不太重要，我就要这个" → Edit 弱（不会调整约束）
```

### 4.5 Synthesize（整合决策）

**探针设计**：决策前确认替代决策后追问

```
用户说"选E"
         ↓
Agent A："好的，你选了E。这套满足要求（...），但有一点（楼层是低层）你确定吗？"
         ↓
观察用户是否修改选择 → Synthesize 信号
```

## 五、MBTI 各维度探针设计

### 5.1 Relation（工具型/伙伴型）

**观察行为模式**：

| 行为 | 倾向 |
|------|------|
| "帮我查一下"、"给我推荐一个" | Instrumental（工具型） |
| "你觉得呢"、"我们来分析一下" | Collaborative（伙伴型） |
| 说完需求就不说话了 | Instrumental |
| 说完需求还继续讨论 | Collaborative |

**探针0（开场）**：
> Agent A：复述约束 + "有什么偏好或要求需要我帮你注意的吗？"

- 用户简短回答（"没了"）→ Instrumental
- 用户详细讨论需求 → Collaborative
- 用户反问 AI 意见 → Collaborative

### 5.2 Workflow（框架型/探索型）

**观察**：用户是先定规则还是边看边调

```
展示6个房源卡片（仅基本信息）
         ↓
观察：
- 用户是否先说筛选条件（"先看价格5000以下的"）→ Framed
- 用户是否一个个看过去边看边想 → Exploratory
```

**主动探针**：
> "要不要我帮你按条件筛选一下？" → 框架型会说"好，按价格排序"

### 5.3 Epistemic（审计型/信任型）

**探针设计**：元认知探针

```
Agent A（主动示弱）："这套我不太确定装修是简装还是精装，你方便再确认一下吗？"
         ↓
观察：
- 用户去确认 → Auditing（愿意花精力核实）
- 用户说"无所谓" → Trusting
```

### 5.4 Repair Scope（全局/局部）

**触发条件**：最优解不存在时

```
Agent A："所有房源都满足不了所有要求，必须放弃一个"
         ↓
观察：
- "那楼层不重要，放弃楼层要求" → Global-Reframing
- "那找一个楼层尽量高一点的" → Local-Refinement
```

## 六、信号冲突处理

用户可能在同一维度上表现矛盾：

```
用户说："你自己觉得呢" → Collaborative
然后说："但我不信，你查一下数据" → Auditing
```

**处理方式**：记录所有信号，不强制归一化到单一值。

```json
{
  "MBTI.Epistemic": {
    "value": 55,
    "confidence": 0.4,
    "signals": [
      {"signal": "Collaborative", "weight": 0.6},
      {"signal": "Auditing", "weight": 0.4}
    ]
  }
}
```

## 七、待讨论问题

1. **路线选择**：让 AI 自己决定探针 vs 行为驱动按需被动触发

2. **探针密度**：一个对话中 Agent A 应该有多少比例的回复包含探针载体

3. **开场探针**：是否需要结构化（固定话术树）还是 Agent A 自由发挥

4. **MBTI 多场景**：单场景是否足够稳定测出 MBTI，还是需要多场景交叉验证

---

## 八、核心结论

> **探针应该是 AI 的自然行为，而不是外部插入的测量工具。**
>
> Agent B 的角色是**异步分析**，不是**实时指挥**。
>
> 规则引擎适用于状态管理和流程控制，但不应参与探针信号的生成过程。
